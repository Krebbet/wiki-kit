# MCP for multi-agent systems (framework paper)

April 2025 framework paper proposing application-layer architectural patterns for building multi-agent systems on top of MCP, positioned as a downstream complement to the [[mcp-infrastructure]] roadmap (which addresses the protocol/transport layer). The paper's lasting contribution is *vocabulary* — a typology of the **context retention problem**, a four-layer reference architecture, and six MCP server patterns. Its empirical section should be treated with substantial skepticism: numbers are uniformly positive against unnamed baselines with no reproducible setup, and the attribution of the predecessor "Krishnan 2025" work is inconsistent in-text vs. in the bibliography. The paper is best used as architectural framing, not as a source of measured outcomes.

## The context retention problem (four manifestations)

The most cleanly transferable contribution. The paper enumerates four distinct ways context fails to persist across multi-agent collaboration:

1. **Discontinuity across agent boundaries** — information gathered or generated by one agent doesn't transfer to others; knowledge gaps and redundant effort.
2. **Temporal discontinuity** — agents lose awareness of past interactions and decisions; cannot build on prior work or maintain consistent approaches.
3. **Contextual prioritization failures** — without relevance mechanisms, agents are overwhelmed with information or miss critical elements.
4. **Cross-modal context integration** — agents operating across text, images, and structured data struggle to integrate diverse sources into coherent understanding.

This typology is more granular than [[memory-architectures]]'s general write-manage-read framing — it specifically names the *multi-agent* failure modes that single-agent memory designs don't address.

## Architectural patterns

- **Four-layer reference architecture**: Agent Runtime → Context Management Layer (MCP servers) → Coordination Framework → External Integration Layer + Security/Governance Layer. Explicit separation of concerns largely absent from typical MAS designs.
- **Six MCP server patterns**: Adapter, Composite, Proxy, and Embedded server styles for different deployment contexts; subscription patterns for live context updates; batch retrieval for initialization; cross-agent resource access via MCP's resource primitive.
- **Context prioritization framework**: multi-dimensional relevance modelling combining topical, temporal, user-specific, and task relevance; collaborative filtering for social signals; task-based context framing.
- **Long-term persistence with forgetting**: hierarchical memory (working → episodic → semantic) mapped to MCP tool primitives; explicit "forgetting strategies" for memory optimisation under volume pressure.
- **Coordination protocols via MCP**: structured message formats as substrate; capability advertisement and discovery through MCP interfaces; asynchronous coordination preferred over synchronous (paper claims 37.8% throughput gain).
- **Market-based resource allocation** (in the DPSS case study only): virtual currency mechanisms, value-based bidding, futures/options for resource access — implemented via MCP tool primitives.

## Three case studies (illustrative architectures)

- **Enterprise Knowledge Management System (EKMS)** — hierarchical seven-agent-type system over a hypothetical 50,000-employee multinational; demonstrates multi-level context hierarchy, temporal validity tracking, and cross-modal integration.
- **Collaborative Research Assistant (CRA)** — peer-based seven-role system for interdisciplinary research teams; cross-disciplinary ontology alignment, hypothesis tracking, reproducibility metadata.
- **Distributed Problem-Solving System (DPSS)** — dynamic-role seven-type system for engineering teams; market-based coordination, versioned solution components.

These are *designs*, not deployments. The paper does not report production rollouts of these systems.

## Empirical section — treat with caution

The paper's quantitative claims are extensive but methodologically thin:

- Long-horizon context continuity: 83.7% (MCP) vs 42.3% (baseline) — biggest delta in the paper.
- Cross-agent context transfer: 79.4% preservation vs 45.7%.
- Communication volume: 47% reduction vs ad-hoc coordination.
- Coordination scaling: O(n log n) vs O(n²) claimed.
- Vs human teams: 4.7× faster, 83% of human quality on average.
- Ablations: removing persistence drops long-horizon performance 41.7%; removing cross-agent context sharing drops collaborative task performance 38.5%.

**Why these are flagged**: baselines are named only generically ("traditional retrieval-augmented generation," "standard agent frameworks") without reference systems, the numbers are uniformly clean and uniformly positive, no datasets are released, and there is no third-party replication. These figures should not be cited as evidence of MCP-based MAS performance — only as the paper's self-reported claims. The architectural vocabulary holds up; the measurements do not.

## Self-documented limitations (Section 8)

- Communication overhead degrades beyond ~1,000 tightly-coupled agents.
- Context volume pressure at ~10⁷ distinct elements.
- Real-time performance constrained at sub-100ms latency requirements.
- Legacy system compatibility requires custom adapter development.
- Versioning/evolution of MCP implementations in long-lived deployments.
- Differential privacy mechanisms introduce utility/privacy trade-offs not yet resolved.
- Auditability and responsibility attribution in complex chains.
- Cross-jurisdiction regulatory compliance tooling immature.
- Heterogeneous deployment (edge, mobile, embedded) not well-supported.

## Provenance caveat

The paper cites "Krishnan (2025)" in-text as the predecessor work establishing the *disconnected models problem*; the corresponding bibliography entry is attributed to "Jiang, A. H., Shen, K., Xu, Z., Wu, Y., Vernon, Z., & Huang, D. (2024)". This in-text vs. bibliography mismatch is a credibility flag worth recording. Authorship of the present paper is not clearly attributable from the captured HTML.

## Relationship to neighbouring wiki pages

- **vs [[mcp-infrastructure]]** (the 2026 roadmap): downstream and orthogonal. The roadmap addresses transport, async lifecycle, governance, and enterprise readiness at the protocol layer; this paper assumes MCP 1.0 as given and proposes application-layer patterns on top. There is no direct conflict.
- **vs [[cognitive-fabric-nodes]]**: directly contrary architectural philosophies. CFN is *middleware-as-substrate* — memory at the network layer, message interception. This paper is *application-layer client-server* — MCP servers as passive endpoints, coordination inside the agent collective. The two could theoretically coexist (CFN intercepting calls to MCP servers) but their mental models are incompatible.
- **vs [[memory-architectures]]**: this paper's hierarchical memory mapping (working → episodic → semantic) instantiates the survey's *hierarchical virtual context* family for multi-agent settings.

## Why it matters

- **Vocabulary contribution.** The four-manifestation typology of context retention failures is the cleanest framing in the wiki for what specifically goes wrong when context crosses agent boundaries.
- **Application-layer counterpoint to mcp-infrastructure.** Sets up the protocol-vs-application split: roadmap items live one layer down; this paper lives one layer up; both will need to mature for production MAS to work.
- **Honest about its limits.** The Section 8 limitations list is unusually candid for a framework paper — the failure thresholds (~1k agents, ~10⁷ context elements, sub-100ms latency) are concrete and useful as planning anchors even if the empirical claims aren't.
